# Document Intelligence Trial Pack — Docling（Windows 本机试投产包）

## 目标
用最小真实样例，验证更强本地文档智能解析能力是否值得进入主工作流。

本试验包重点不是“能不能装上”，而是验证它是否能稳定提升这些复杂真实任务：
- 复杂 PDF / 扫描件 / 报告 / 多栏文档读取
- 表格抽取后继续整理到 Markdown / 表格 / Office 流程
- 研究材料转结构化证据包
- 文档处理前的可消费输入准备

## 适用触发条件
当任务满足以下任一情况时，优先考虑按本包做一次试验：
1. 现有 `office-local` / `pdf` / 手工整理链路能读，但结构乱、顺序乱、表格难复用
2. 文档是扫描件、图片型 PDF、多栏排版、混合表格或复杂版式
3. 下游还要进入 `document-task-routing`、`research-evidence-pack`、`office-output-proof`、简历/Office/表格处理链路
4. 同类复杂文档开始重复出现，值得判断是否升级成长期能力

## Trial 样例篮子
默认准备 3 个真实样例，不追求多，追求能覆盖典型难点：

### Sample A — 普通 PDF
- 目标：确认基础阅读顺序与 Markdown 输出质量
- 通过重点：标题层级、段落顺序、列表结构是否自然

### Sample B — 复杂表格 / 多栏 PDF
- 目标：确认版面理解与表格结构提取是否优于现有链路
- 通过重点：
  - 多栏内容是否串栏
  - 表格是否还能被下游继续消费
  - 表头、单元格、行列关系是否可用

### Sample C — 扫描件 / 图片型文档
- 目标：确认 OCR + 结构化输出是否真的有增益
- 通过重点：
  - 中文识别可读性
  - 段落切分是否还原
  - 是否明显减少人工清洗

## 统一验收维度
每个样例都按这 6 项打结论，不只看“输出成功”：

1. **输出生成**
   - 是否稳定生成 Markdown / JSON / 表格结果
2. **阅读顺序**
   - 是否比现有链路更少错位、串栏、错段
3. **表格可消费性**
   - 表格能否继续进入表格整理 / Office / 结构化链路
4. **OCR 可用性**
   - 扫描件是否达到“可继续做事”而不只是“勉强能看”
5. **人工清洗成本**
   - 相比现有链路，后处理是否明显减少
6. **Windows 可维护性**
   - 安装、依赖、运行、复现是否足够稳

## 结论分级

### Invest now
仅当以下条件大体同时成立：
- 至少 2/3 样例有明显收益
- 其中至少 1 个复杂样例（表格/扫描件）明显优于现有链路
- 输出可直接喂给下游任务，而不是还要大洗一遍
- Windows 维护成本可接受

### Trial only
适用于：
- 有提升，但还不稳定
- 只在部分样例明显更好
- 需要更多真实场景确认 ROI

### Defer
适用于：
- 输出质量没有明显超过现有链路
- 安装/依赖/运行过脆
- OCR / 表格 / 中文效果不稳定
- 下游仍需要大量人工重构

## 下游接法
如果结论是 **Invest now**，按这个顺序推进：
1. 产出 `Capability Proof Pack`
2. 决定进入哪条长期路径：
   - 新增轻量 skill
   - 接到 `document-task-routing.md`
   - 作为 `office-local` 的增强分支
3. 给出最小调用入口与回退路径

如果结论是 **Trial only**：
- 保留为候选增强能力
- 只在复杂 PDF / 表格 / 扫描件任务里按需启用
- 暂不进入默认主链路

如果结论是 **Defer**：
- 继续使用现有 `office-local`、`pdf`、`document-task-routing` 链路
- 等复杂样例密度变高或工具成熟后再复查

## 复用规则
以后对“新文档解析能力”默认不要问：
- 它支不支持很多格式？
- 官网写得强不强？

而要先问：
- 它能不能把复杂真实文档稳定变成下游可消费输入？
- 它是否真的减少人工清洗？
- 它在 Windows 本机是否值得长期维护？

## 本次固定下来的方向
把“复杂文档 -> 结构化输出 -> 下游工作流消费”列为继续投入的进化方向。

优先观察的真实任务信号：
- 复杂 PDF 表格抽取反复卡住
- 扫描件 OCR 结果不能直接用
- 研究/报告文档需要频繁人工重排
- 文档提取结果无法稳定进入简历/Office/证据包/表格链路
