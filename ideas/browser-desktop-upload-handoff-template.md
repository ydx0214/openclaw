# Browser/Desktop Upload Handoff Template

## Why this exists

复杂真实任务里，浏览器自动化最容易在“点击 Upload 后弹出 Windows 文件选择框”这一段断开。问题通常不是不会点按钮，而是浏览器上下文、桌面窗口、目标文件路径、上传后验收这 4 件事没有被统一收口。

这个模板用于把“网页上传动作”稳定拆成可执行的交接步骤，优先服务：

- 官网求职投递上传简历/附件
- 网站表单上传 PDF / DOCX / 图片
- 浏览器流程中需要调用 Windows 文件对话框的场景

## Trigger

当任务同时满足以下两点时使用：

1. 当前主流程在浏览器里
2. 文件上传需要经过 Windows 原生文件选择框，或浏览器上传控件行为不稳定

## Goal

把上传步骤从脆弱的一次性点击，变成 `preflight -> handoff -> verify` 的稳定流程。

## Preflight

在点击上传前，先明确这 5 个点：

1. **目标文件路径**
   - 文件存在
   - 文件名和扩展名正确
   - 如果有多个候选文件，先选定唯一目标
2. **页面上下文**
   - 当前在哪个网站/表单/步骤
   - 哪个字段要上传什么类型的文件
3. **上传方式**
   - 优先浏览器直接 `upload` / `fill file input`
   - 不稳定时再切到桌面文件对话框接管
4. **验收信号**
   - 页面出现文件名
   - 页面出现 uploaded / complete / ready 之类状态
   - 下一步按钮解锁
5. **覆盖风险**
   - 是否会替换现有附件
   - 是否需要保留旧文件

## Default Handoff Flow

### Route A: Browser-native first

1. 定位上传控件或 input[type=file]
2. 直接传入文件路径
3. 回到页面核验上传结果

适用：页面允许直接操作文件 input，且不会弹系统对话框。

### Route B: Browser -> Windows dialog handoff

1. 在浏览器中确认上传按钮/区域
2. 点击触发文件选择框
3. 切换到桌面自动化流程：
   - 确认文件对话框已出现
   - 优先用语义方式定位文件名输入框/打开按钮
   - 填入绝对路径
   - 执行打开
4. 回到浏览器页面
5. 重新检查页面是否显示上传成功信号

适用：浏览器原生上传不可用，或点击后必然弹 Windows 文件选择框。

## Verification Checklist

上传后至少验证其中 2 项，优先第 1 和第 2 项：

1. 页面已显示目标文件名
2. 页面状态变化为已上传/可继续
3. 预览区域出现新附件
4. 网络/页面步骤进入下一阶段
5. 若是求职网站，附件区和申请步骤都未报错

## Failure Patterns

### 1. 以为上传了，其实只是打开了对话框

修正：必须在页面侧再次验证文件名或状态变化。

### 2. 文件路径没先确认，弹窗里临时找文件

修正：上传前先做文件存在性检查，尽量使用绝对路径。

### 3. 桌面点击成功，但浏览器仍停在旧状态

修正：把页面验收作为独立步骤，不以桌面动作成功代替上传成功。

### 4. 多文件候选时临时犹豫

修正：在 preflight 就固定“本次上传文件”。

## Reusable Report Snippet

- 上传目标：`<字段/用途>`
- 文件：`<绝对路径>`
- 方式：`browser-native` / `browser->windows-dialog`
- 验证结果：`<文件名已显示 / 状态已变更 / 下一步已解锁>`
- 剩余人工确认：`<如无可写 无>`

## Suggested Long-term Rule

对所有“浏览器流程里带附件上传”的复杂任务，默认把上传视为一个独立子任务，而不是普通点击；先固定文件、再选择接管方式、最后做页面侧验收。
